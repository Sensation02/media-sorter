# Anti-Patterns

This file is the codified memory of mistakes already made on this project. Every PR review can add new entries here.

## Format

Each entry has the structure:

- **AP-XXX: `<Short name>`**
- **Problem:** description of the bad pattern
- **Why it's bad:** consequence — what breaks, what becomes hard, what gets confusing
- **Fix:** correct approach
- **First seen in:** PR number or commit SHA

Identifiers are sequential (`AP-001`, `AP-002`, ...). Never reuse or renumber — once assigned, an AP-XXX is permanent.

## Entries

- **AP-001: `Tauri command registration via re-export`**
- **Problem:** Registering a `#[tauri::command]` function in `tauri::generate_handler![...]` through a re-exported path (e.g. `scanning::pick_source_dir`) when the actual definition lives deeper (`scanning::command::pick_source_dir`).
- **Why it's bad:** The `#[tauri::command]` proc-macro generates two sibling macros next to the function (`__tauri_command_name_*`, `__tauri_cmd__*`). A `pub use module::name` only re-exports the named item — the function — and leaves the sibling macros behind. `generate_handler!` then does string replacement on the registration path: it takes the last segment and rewrites it to `__tauri_command_name_*`. When the path points at a re-export, the macro lookup resolves into the wrong module and the build fails on a fresh compile.
- **Fix:** Register in `generate_handler!` with the full path to where the function is actually defined: `scanning::command::pick_source_dir`. Keep `pub use command::name` re-exports only for ordinary imports from sibling modules.
- **First seen in:** PR #7 (commit `c7c503d`, EPIC-03)

- **AP-002: `Trusting third-party From<io::Error> mapping`**
- **Problem:** Calling `external_fn(path)?` at a third-party boundary — where `external_fn` opens the file internally — and assuming the resulting `io::Error` will land in the "I/O variant" of that crate's error enum.
- **Why it's bad:** Not every crate follows that convention. Example: `nom-exif` has `impl From<io::Error> for nom_exif::Error { fn from(e) -> Self { ParseFailed(e.into()) } }`, so an `io::Error` arrives as `ParseFailed`, not `IOError`. If our code matches on `IOError` to classify I/O failures, real I/O problems (missing file, permission denied) silently slip through as "format not recognised" and we return an empty result instead of `AppError::Io`.
- **Fix:** Keep the I/O boundary in our own code. Open the file ourselves (`File::open(path).map_err(...)`) **before** handing the handle to the third-party crate. The `io::Error` then flows through **our** `From<io::Error> for AppError` impl (where it lands in `AppError::Io`), and any error coming back from the third-party is guaranteed to be about format, not I/O.
- **First seen in:** PR #7 (commit `92ea1ec`, EPIC-03)

- **AP-003: `Bypassing Tailwind v4 theme tokens with bg-[var(--color-X)]`**
- **Problem:** Writing arbitrary-value classes like `bg-[var(--color-bg)]`, `text-[var(--color-fg-1)]`, `border-[var(--color-border)]` in JSX even though the project's `@theme inline` block in `src/index.css` already registers these as Tailwind tokens — which means `bg-bg`, `text-fg-1`, `border-border` are auto-generated and equivalent.
- **Why it's bad:** Bundle pays for every arbitrary value as a unique class. The Tailwind VSCode plugin shows a warning (`The class \`bg-[var(--color-bg)]\` can be written as \`bg-(--color-bg)\``). IntelliSense, autocompletion, and rename-symbol stop working inside `[var(...)]`. Worst of all, the pattern signals to future contributors that the design-system tokens are not load-bearing — which encourages more drift. By PR #43 there were 103 such uses across 20 files.
- **Fix:** Always prefer the token class generated by `@theme inline`. If you genuinely need an inline CSS variable not registered in `@theme`, use the Tailwind v4 shorthand `bg-(--var-name)` (no `var()` wrapper, no square brackets).
- **First seen in:** PR #43 (EPIC-12)

- **AP-004: `Magic pixel values as arbitrary Tailwind classes`**
- **Problem:** Using `text-[13px]`, `text-[10.5px]`, `h-[3px]`, `tracking-[1px]`, `w-[30px]`, `translate-x-[14px]` etc. as one-off arbitrary classes when the same value recurs across many files (e.g. `text-[13px]` appeared 19 times, `text-[10.5px]` 8 times).
- **Why it's bad:** No single source of truth. Subtle drift is invisible: in PR #43 the `Stat` component used `text-[10px]` + `tracking-[0.6px]` while every other "eyebrow" label used `text-[10.5px]` + `tracking-[1px]` — a near-invisible inconsistency that only shows up at specific monitor densities, and was never noticed by review. The typographic/geometric scale lives in heads instead of in code.
- **Fix:** Define semantic tokens in `@theme inline` (e.g. `--text-eyebrow: 10.5px; --text-meta: 11.5px; --text-body: 13px; --text-display: 28px`) and use the generated classes (`text-eyebrow`, `text-meta`). Same applies for component geometry that recurs: `--switch-width`, `--select-trigger-height`, `--progress-thickness`. A magic px value is acceptable only when it's truly unique to one element and unlikely to recur.
- **First seen in:** PR #43 (EPIC-12)

- **AP-005: `Unicode characters as icons`**
- **Problem:** Using glyphs like `"✦"`, `"◷"`, `"⚙"`, `"▸"`, `"↶"`, `"▣"`, `"→"` as visual icons in JSX, often typed as `string` (e.g. `icon: "✦"` in a sidebar items table).
- **Why it's bad:** (a) stroke width and weight depend on the system font and change unpredictably across OS/locales; (b) no automatic `aria-hidden`, so screen readers may announce the literal name of the character; (c) sizing requires `text-[Npx]` (an [AP-004](#AP-004) trap); (d) the design system feels accidental rather than authored; (e) some glyphs render as `□` on older fallback fonts. The codebase already depended on `lucide-react`, used in only 3 spots.
- **Fix:** Migrate true icons to `lucide-react`. Centralise icon imports in `src/features/sort/constants/icons.tsx` so the project has one inventory. Type icon references as `LucideIcon` (component reference), render as `<Icon className="h-X w-X" aria-hidden />`. **Typographic glyphs are not icons** — `·` (middot), `…` (ellipsis), `—` (em dash) remain inline text characters; they participate in baseline alignment and are part of the copy, not the iconography.
- **First seen in:** PR #43 (EPIC-12)

- **AP-006: `Co-located components in a single file`**
- **Problem:** Helper components defined inside the same `.tsx` file as their primary component. Examples observed in PR #43: `Placeholder` inside `PreviewTree.tsx`, `HistoryRow` + `HistoryStats` + `Centered` (three!) inside `HistoryScreen.tsx`, `SettingsForm` inside `SettingsScreen.tsx`.
- **Why it's bad:** Directly violates CLAUDE.md: _"every UI component lives in its own file. Never co-locate two or more components in a single file, including small private helpers."_ Co-location prevents testing helpers in isolation, mixes file responsibilities, and lets unrelated logic accrete inside one module without anyone noticing. Once a file has three components, it usually keeps growing.
- **Fix:** Each component in its own file inside the feature folder. `HistoryScreen/` becomes a folder containing `HistoryScreen.tsx`, `HistoryRow.tsx`, `HistoryStats.tsx`, `Centered.tsx`, and a barrel `index.ts`.
- **First seen in:** PR #43 (EPIC-12)

- **AP-007: `Composition component mixing UI, state, side effects, and mappers`**
- **Problem:** A top-level composition component (such as `SortApp.tsx`) holding all of: JSX composition, multiple `useState` + `useReducer` + `useEffect` + `useCallback` orchestrating a state machine, DTO→view-model mappers (`toSortDone`, `revertSummary`), pure routing helpers (`resolveScreen`, `preferredDefaultRule`), and hardcoded label tables (`TOOLBAR_TITLE`, `TOOLBAR_STATUS`, `TOOLBAR_SUBTITLE`). In PR #43 audit, `SortApp.tsx` was 309 lines and could not be tested in isolation.
- **Why it's bad:** Untestable. Every mapper change forces re-render of the whole tree. Mental model breaks — the file is "App" but does mapping. Refactoring later becomes high-risk because changes are entangled.
- **Fix:** Composition file stays JSX-only. Extract: (a) orchestration hook (`hooks/use-X-orchestration.ts`) holding state machine + handlers, (b) DTO mappers (`mappers/`), (c) constants (`constants/`), (d) pure routing helpers (`screens/resolve-X.ts`). Each piece becomes independently testable.
- **First seen in:** PR #43 (EPIC-12)

- **AP-008: `Duplicate string literals for a typed union`**
- **Problem:** A literal-string union type like `SortScreen = "setup" | "progress" | "done" | "history" | "settings"` is declared once in `types/`, but the string literals are inlined dozens of times throughout the codebase: `setScreen("progress")`, `if (screen !== "progress")`, `case "done":`, etc. In PR #43, `SortApp.tsx` alone referenced these literals 15 times.
- **Why it's bad:** Single source of truth violated. Adding a new variant requires grep-and-pray. Renaming a variant requires manual sweep (auto-rename of a TS string is not refactor-safe). `Object.values(SORT_SCREEN)` is impossible — you cannot iterate a union type at runtime.
- **Fix:** Use the `as const` object pattern with a derived type:

    ```ts
    export const SORT_SCREEN = {
        setup: "setup",
        progress: "progress",
        done: "done",
        history: "history",
        settings: "settings",
    } as const;

    export type SortScreen = (typeof SORT_SCREEN)[keyof typeof SORT_SCREEN];
    ```

    Then write `setScreen(SORT_SCREEN.progress)`, `screen !== SORT_SCREEN.progress`, and iterate with `Object.values(SORT_SCREEN)`.

- **First seen in:** PR #43 (EPIC-12)

- **AP-009: `as TypeId cast at an untyped boundary`**
- **Problem:** Writing `onChange(next as SortRuleId)` where `next: string` arrives from an external API such as Radix `onValueChange`. The cast bypasses any guarantee that `next` is actually a valid `SortRuleId`.
- **Why it's bad:** `as` is a compile-time lie. If a future caller passes an unknown id (e.g. a stale value from a settings migration), it silently propagates and crashes downstream in a far-away component, with a confusing stack trace. CLAUDE.md explicitly forbids lazy escape hatches in typing.
- **Fix:** Define a type guard next to the type definition:
    ```ts
    export function isSortRuleId(value: string): value is SortRuleId {
        return value in SORT_RULE_IDS;
    }
    ```
    At the boundary, validate and decide explicit behavior on miss (no-op, log + toast, or fall back to a default).
- **First seen in:** PR #43 (EPIC-12)

- **AP-010: `Inline <button> bypassing the component library`**
- **Problem:** Writing `<button className="inline-flex … bg-[var(--color-primary)] …">` inside a component (the audit found one such inline button in `ErrorBoundary.tsx`) instead of using the project's `<Button variant="primary">` primitive.
- **Why it's bad:** Consistency drift. When `<Button>` evolves — adds a focus ring style, changes hover behavior, fixes a disabled-cursor bug — the inline copy doesn't move with it. Search-and-replace cannot find the inline copy by class signature.
- **Fix:** Always use `<Button>` (or whatever primitive exists for that role). If the variant you need is missing, **add it to `buttonVariants` cva** rather than inlining a one-off.
- **First seen in:** PR #43 (EPIC-12)

- **AP-011: `Duplicate visual primitives ("eyebrow", "screen frame")`**
- **Problem:** Identical or near-identical class combinations repeated in many files. PR #43 found the "eyebrow" pattern (`font-mono text-[10.5px] uppercase tracking-[1px] text-fg-3 mb-2.5`) duplicated 8 times across screens — with one outlier in `Stat` that drifted to `text-[10px]` + `tracking-[0.6px]`. The "screen frame" wrapper (`flex flex-col h-full > flex-1 overflow-y-auto px-7 py-7 > footer h-12 border-t …`) was duplicated in 4 screens.
- **Why it's bad:** A design tweak (e.g. "make eyebrow 1px larger") must be applied to N files synchronously; missing one creates a silent regression. Drift between copies is invisible until pixel-comparing screens side by side.
- **Fix:** Extract reusable primitives once the pattern is duplicated **three or more times** (the CLAUDE.md "three similar lines beats premature abstraction" rule). Examples in PR #43: `<Eyebrow tone="default|warning|destructive">`, `<ScreenFrame footer={…}>`. Keep the primitive minimal — props are a hard constraint, not a feature menu.
- **First seen in:** PR #43 (EPIC-12)

- **AP-012: `Dead code surviving via unused exports`**
- **Problem:** Components, types, or constants exported from a module but never imported anywhere. Examples in PR #43: `CardHeader`, `CardContent`, `CardFooter` exported from `card.tsx` with zero consumers; `DEFAULT_SETTINGS` exported from `features/sort/constants.ts` with zero consumers (settings now come from the backend); the `SortSettings` TypeScript type with zero consumers; `features/sort/components/index.ts` barrel where no caller imports from it (every consumer imports directly from the subfolder).
- **Why it's bad:** Maintenance burden with no payoff. New contributors waste time wondering when to use the export. Refactors must keep proving it doesn't matter. TS does not flag this without `ts-prune` or similar in CI.
- **Fix:** Delete on sight. If a planned future use exists, note it in the related epic spec — never in code with `// TODO use later`. If you add a barrel, verify at least one consumer imports through it; otherwise delete the barrel.
- **First seen in:** PR #43 (EPIC-12)

- **AP-013: `Misleading "LEGACY" naming for active code`**
- **Problem:** A constant named `LEGACY_SORT_SETTINGS` actively passed to `startSort(plan, LEGACY_SORT_SETTINGS)` on every sort. The name implies a temporary/migration artifact, but the value is load-bearing.
- **Why it's bad:** "Legacy" is a contract about lifetime. Future cleanup will delete it in good faith and break the running app. Wrong names mislead grep, mislead reviewers, and waste the next contributor's time deciding whether they're allowed to remove it.
- **Fix:** Name things by what they are. If the value is intentionally fixed at the call site until the UI surfaces the flags, name it `IMMUTABLE_SORT_FLAGS` or `SORT_FLAGS_UI_NOT_YET_WIRED` and pair it with a `TODO` referencing the tracking epic. If it's truly legacy, delete it.
- **First seen in:** PR #43 (EPIC-12)
