# Artifacts: HTML for humans

> Convention for ephemeral HTML files generated by agents to help the human reader understand a task, design, plan, or change visually — without producing yet another long Markdown file.
> Companion to `orchestration.md`, `implementation.md`, and `delivery.md`.

---

## Purpose

`.artifacts/` is local-only scratch space (gitignored). It exists so an agent can produce a richer, denser, more readable artefact than a Markdown file when the goal is to **show the human reader something once**, not to add a durable contract to the repo.

Inspiration: Thariq Shihipar, *"Using Claude Code: The Unreasonable Effectiveness of HTML"* (2025). The idea is **scoped down** here: HTML is used for ephemeral human-facing artifacts only. Specs, governance, CHANGELOG, and process docs stay in Markdown.

## When to use HTML vs Markdown

| Audience                                  | Lifetime  | Format   | Where         |
| ----------------------------------------- | --------- | -------- | ------------- |
| Agent (this file, future agents, scripts) | Durable   | Markdown | `docs/`, root |
| Human (one-shot, "show me")               | Ephemeral | HTML     | `.artifacts/` |

Decision rule: if a doc is grep-ed, diff-ed, parsed, or referenced from code — **Markdown**. If a doc is opened in a browser, scrolled once, then closed — **HTML in `.artifacts/`**.

## Use cases

- **PR explainer** — visual diff with severity-coded annotations and a focused walkthrough; link attached in the PR description.
- **Design exploration** — N UI variants side-by-side with tradeoff labels, before committing to one direction.
- **Implementation plan** — rich plan with mockups, data-flow SVG, and code snippets, before a long Strategy B/C/D task.
- **Research summary** — synthesis from codebase, git history, and external sources, easier to read than a 300-line Markdown discovery.
- **Custom editing UI** — throwaway tool for one piece of data (config diff, conflict-resolution rules, prompt tuner) with a "copy as JSON / copy as prompt" export back into the agent.

## Storage and lifecycle

- **Folder:** `.artifacts/` at repo root. Gitignored.
- **Filename:** `<topic>-<YYYY-MM-DD>.html` (e.g. `pr-42-explainer-2026-05-09.html`).
- **Lifetime:** until the human reader no longer needs it. No retention policy — delete freely; `git clean -fdx` will wipe the folder.
- **Sharing:** manual. Open locally in a browser; if a URL is needed, upload to S3 / Gist / a `gh-pages` branch, never to the main repo history.

## Hard rules

1. **Never** put durable contracts (specs, governance, CHANGELOG, workflow docs) in `.artifacts/`.
2. **Never** import `.artifacts/` content from production code (`src/`, `src-tauri/`).
3. **Never** reference `.artifacts/` paths from durable Markdown docs — those paths are local-only and disappear on `git clean`.
4. Generation cost: HTML takes ~2–4× the tokens of equivalent Markdown (per the source article's FAQ). Use HTML when visual richness genuinely helps; default to Markdown otherwise.

## Anti-pattern: HTML maximalism

Wrapping every plan in HTML wastes tokens, slows iteration, and adds zero value when the content is a bulleted list. Default to a chat reply or a short Markdown file. Reach for HTML in `.artifacts/` only when at least one of the following is true:

- Multiple visual options to compare side-by-side
- Diagrams, charts, or color-coded data that ASCII cannot represent
- Interactive controls (sliders, toggles, drag-and-drop) that feed back into a prompt
- The human reader explicitly asks for a visual artifact

## Out of scope

- **In-app user-facing help / onboarding** — that is production code in the Tauri webview, with separate concerns (i18n, bundle size, sandboxing). It does NOT live in `.artifacts/`.
- **Public documentation site** — if that need ever arises, it gets its own folder and its own discussion.
